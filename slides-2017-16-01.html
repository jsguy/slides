Framework or library

## With great power... * If you choose a library, it is your _[responsibility](resource/img/with-great-power-weeee.gif)_ to implement [the other parts](https://en.wikipedia.org/wiki/Model%E2%80%93view%E2%80%93controller) * If you choose a framework, it will _enforce_ a particular implementation
## How to evaluate? * **Objective** - measurable attributes for a library/framework, eg: production install base * **Subjective** - opinion based attributes of a library/framework, eg: developer sentiment
## Objective considerations * **Popularity** - eg: how many stars does it have on Github? * **Integrations** - eg: how many offical integrations are available with other frameworks * **Level** of control - eg: browser compatibility * **Overheads** - eg: how large is a typical CSS/JS download for the library/framework
## Subjective considerations These are based on a small team * **Maintainable** - how easy is it for a developer to modify and keep the UI up to date whilst adhering to coding standards? * **Expressive** - how quickly can a developer get a new solution out the door? * **Performant** - how well does the UI perform in complex situations?

A general observation about UI frameworks

  • If a framework is Maintainable and Performant, it won't be Expressive
  • If a framework is Maintainable and Expressive, it won't be Performant
  • If a framework is Performant and Expressive, it won't be Maintainable
